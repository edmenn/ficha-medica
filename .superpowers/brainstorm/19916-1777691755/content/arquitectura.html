Arquitectura del sistema — 3 enfoques

Elegí el que mejor se adapta a tus necesidades

A

Next.js + Supabase full-stack

Frontend y API en Next.js (Vercel), base de datos + auth + storage en Supabase. La API key del usuario se guarda encriptada en la DB y las llamadas a IA se hacen desde el servidor (API Routes de Next.js). Stack: TypeScript, Next.js 14, Supabase, PostgreSQL.

Pros

  • 100% gratis en free tier
  • Un solo repo, fácil de mantener
  • Deploy automático en Vercel
  • API key nunca llega al cliente

Cons

  • Vercel free: timeout de 10s en API routes (raro para análisis de imágenes)
  • Cold starts ocasionales

⭐ Recomendado

B

Next.js + Supabase + Edge Functions

Igual que A, pero el procesamiento de IA se mueve a Supabase Edge Functions (Deno). Más robusto para tareas de larga duración. Stack: TypeScript, Next.js 14, Supabase, Edge Functions.

Pros

  • Sin límite de timeout de Vercel para IA
  • Lógica de IA colocada junto a la DB
  • 500K invocaciones gratis/mes

Cons

  • Deno runtime en Edge Functions (diferente a Node)
  • Más complejo de debuggear localmente
C

Next.js frontend + Backend Python (FastAPI)

Frontend en Next.js/Vercel, backend separado en Python con FastAPI hosteado en Railway o Render. Más potencia para procesamiento de imágenes y futuras integraciones ML. Stack: TypeScript, Next.js, Python, FastAPI, Supabase.

Pros

  • Máxima flexibilidad
  • Fácil agregar procesamiento ML en el futuro
  • Sin limitaciones de runtime

Cons

  • Railway/Render free tier duerme tras inactividad
  • Dos repos / más ops
  • Probablemente overkill para el MVP
